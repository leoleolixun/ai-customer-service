# AI Customer Service Development Rules

本文件是项目开发的强制规则。版本顺序和逐版本验收见
`docs/version-roadmap.md`，V1.0 详细执行计划见
`docs/v1-development-plan.md`，长期产品方向见 `docs/development-plan.md`。

## 1. 当前版本目标优先

- V1.0 只交付与具体业务系统无关的多租户 AI 客服核心。
- 博客知识源接入属于 V1.1，商城实时业务工具属于 V1.2，不得提前写入 V1.0 核心。
- 不在当前版本范围内的功能不得顺手实现，统一进入后续版本 Backlog。
- 每个版本开始前必须固定目标、范围、明确不做、测试矩阵和量化验收。
- 每个版本完成的判断依据是该版本验收结果，不是代码量、页面数量或开发天数。
- 市场访谈、试用数量和付费意愿是产品决策门禁，不替代版本技术验收。

## 2. 架构边界

- V1.0 使用模块化单体：一个 FastAPI API、一个 Worker、一个 PostgreSQL、一个 Redis。
- HTTP 层只负责认证、参数解析、调用 Service 和转换响应。
- Service 负责业务规则和事务边界。
- Repository 负责数据库访问，不接收 FastAPI Request 或 Response。
- Provider 负责 LLM、Embedding、对象存储和外部工具等第三方能力。
- 领域之间通过 Service 接口或持久化事件交互，禁止跨领域直接修改数据表。
- 不提前拆分微服务，不为了模式完整而创建没有实际行为的抽象层。

## 3. API 与契约

- 公共 API 统一使用 `/v1` 前缀。
- FastAPI 路由、Pydantic Schema 和响应模型是 OpenAPI 的单一事实来源。
- 设计新接口时先写请求、响应、权限和错误契约，再实现业务逻辑。
- CI 导出并检查 OpenAPI 快照，不再手工维护一份容易漂移的重复 YAML。
- 错误响应使用 `application/problem+json`，包含稳定错误码和 `request_id`。
- 创建类接口在存在重试风险时支持 `Idempotency-Key`。
- 流式接口使用明确的 SSE 事件协议，事件名称和结束语义必须有契约测试。
- 破坏性 API 变更必须发布新版本，不能静默修改现有字段语义。

## 4. 多租户和权限

- 租户拥有的业务表必须包含 `tenant_id`；应用级资源同时包含 `application_id`。
- `Tenant` 自身和明确的平台级资源不添加伪造的 `tenant_id`；平台/租户共用表必须用 `scope` 和数据库约束保证两种归属互斥。
- `tenant_id` 只能从认证上下文获得，不接受请求体传入的租户身份。
- Repository 的租户查询必须显式接收 `tenant_id`，禁止提供无租户条件的通用查询。
- 数据库唯一约束、Redis Key、对象存储路径、任务消息和向量检索条件都包含租户边界。
- 管理接口同时校验角色、Scope、租户归属和资源状态。
- 每个新增资源必须有跨租户读取、修改和删除测试。
- V1.0 使用应用层强制隔离；关键表增加 PostgreSQL RLS 作为后续加固项，不阻塞核心版。

## 5. 数据库与事务

- 所有 Schema 变更通过 Alembic，禁止在应用启动时使用 `create_all` 修改生产库。
- migration 必须同时在空库和上一版本数据库上验证。
- 时间统一使用 UTC，金额和费用不使用浮点数。
- 外键、唯一约束、索引、删除策略和数据保留期必须在建表时定义。
- Service 明确事务边界，Repository 不自行提交调用方事务。
- 跨进程操作先持久化状态，再投递幂等任务；重要事件使用 Outbox 或等价可靠机制。

## 6. 模型配置与调用

- 业务 Service 不直接依赖某个模型厂商 SDK，只依赖项目定义的 Provider 接口。
- V1.0 分别实现 Fake Chat/Embedding Provider，以及 OpenAI-compatible Chat/Embedding Provider，不逐家复制 SDK。
- 模型 Base URL、模型名、API Key 和预算属于数据库运行配置，通过管理面维护。
- 部署配置只保存数据库、Redis、对象存储和根加密密钥等启动参数。
- 供应商密钥必须加密保存，日志和响应中不得出现完整值。
- 普通单元测试和 CI 使用 Fake Provider；真实模型测试只在显式的手动或发布前任务执行。
- 每次模型调用记录租户、应用、会话、模型、Token、耗时、状态和费用估算。
- 模型输出不能直接决定权限、资金操作或数据归属。

## 7. RAG 与业务工具

- 售后政策、商品说明和帮助文档进入知识库。
- 订单、库存、物流、支付和退款进度通过受控业务工具实时查询。
- 禁止为了方便把动态业务数据定时复制进向量库作为事实来源。
- 文档入库必须版本化，并保存来源、Checksum、解析状态、Embedding 模型和分块策略版本。
- V1.0 检索采用租户过滤后的混合检索：向量召回、关键词召回、结果融合。
- Reranker 只有在评估集证明收益高于延迟和成本时才启用。
- 回答必须携带可验证引用；证据不足、冲突或过期时拒答或转人工。
- 检索内容视为不可信输入，文档内容不得覆盖系统规则和工具权限。
- Embedding 模型更换必须使用新版本重建索引，禁止把不同模型向量混在同一索引中比较。

## 8. 工具调用

- 每个工具先定义 JSON Schema、权限、风险级别、超时、幂等和审计，再允许模型选择。
- V1.0 不实现真实业务工具；从 V1.2 开始只允许自动执行只读工具。
- 写操作、资金操作和敏感数据操作在具备独立版本设计与验收前一律转人工处理。
- 工具服务必须再次校验最终用户身份和资源归属，不能相信模型生成的用户 ID 或订单号。
- 外部 URL 必须进行 SSRF 防护，调用必须签名并防止重放。
- 工具失败不得伪造成功结果，必须明确降级、重试或转人工。

## 9. 后台任务

- 文档解析、Embedding、索引更新和离线评估必须由 Worker 执行。
- 不使用 FastAPI `BackgroundTasks` 承担必须完成的持久任务。
- 任务必须幂等，记录阶段、尝试次数、错误和最终状态。
- 删除或更新文档时，数据库状态、对象文件和向量索引必须最终一致且可重试。

## 10. 安全与隐私

- API Key 只保存不可逆 Hash；需要再次使用的供应商密钥使用根密钥或 KMS 加密。
- 日志不得记录密码、完整 Token、供应商密钥、原始工具凭据和不必要的用户隐私数据。
- 上传文件限制类型、大小、页数和解压体积，V1.0 不处理扫描件 OCR。
- 所有用户输入、知识片段和工具结果都按不可信内容处理。
- 高风险操作不得仅凭 LLM 输出执行。
- 代码库不得提交真实密钥、真实客户文档或生产会话数据。

## 11. 测试规则

- 单元测试覆盖纯业务规则、状态机、分块、融合排名、权限和费用计算。
- 集成测试使用真实 PostgreSQL、pgvector 和 Redis，验证事务、约束和查询行为。
- 契约测试覆盖 OpenAPI、Provider 和 SSE 事件顺序；工具版本再增加工具签名契约。
- 隔离测试覆盖 API、数据库、缓存、对象路径、Worker 和向量检索。
- V1.0 E2E 覆盖 Widget 对话、知识引用、拒答和人工接管；后续版本按版本验收增加博客同步或商城工具场景。
- RAG 变更必须运行固定评估集，禁止只凭人工聊天感觉判断效果。
- 不追求无意义的总覆盖率数字，但核心领域分支和安全边界不得缺少测试。

## 12. 可观测性

- 每个请求产生 `request_id`，跨 API、Worker、模型和工具调用传递。
- 使用结构化日志，错误记录稳定错误码和可定位上下文。
- 至少记录 API 延迟、首 Token 时间、检索耗时、模型费用和转人工率；工具版本增加工具错误指标。
- 不记录模型隐藏推理过程，只保存业务所需的回答、引用、结构化工具选择和审计信息。

## 13. 完成定义

一个功能的开发只有同时满足以下条件才算完成：

- API 契约和权限边界明确。
- 数据迁移、索引和删除策略完整。
- 正常、异常、越权和幂等路径有测试。
- 日志、用量或必要指标可观测。
- OpenAPI 和接入文档同步更新。
- 不引入当前版本范围外的隐性依赖。
- 通过该功能相关的自动化测试和阶段完成检查。

单个功能完成、阶段检查通过或开发环境基线达标，都不等于版本验收通过。版本流程必须按以下顺序执行：

1. 当前版本冻结范围内的所有功能、迁移、测试工具和文档完成。
2. 固定待验收 Commit、构建产物和验收环境。
3. 针对完整版本执行量化验收、回归、安全、性能和恢复演练。
4. 生成并保存该版本验收记录，包括测试命令、结果、评估指标、已知限制和发布结论。

只有第 4 步结论为“通过”时，版本才算完成并允许发布 Tag。

## 14. 提交规则

- 提交格式使用 `<type>(<scope>): <中文描述>`。
- 常用类型包括 `feat`、`fix`、`docs`、`refactor`、`test`、`chore` 和 `build`。
- 每次提交只包含一个清晰意图，不把无关格式化或生成文件混入功能提交。
