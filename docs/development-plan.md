## 2. 产品定位

项目定位为面向国内电商、内容站点和 SaaS 系统，支持私有部署和快速接入的多租户 AI 客服平台。平台长期解决以下问题：

1. 租户管理自己的应用、模型、知识库和客服成员。
2. AI 根据可追溯证据回答，并在证据不足时拒答。
3. 外部内容通过通用内容源同步，不把博客逻辑写进平台核心。
4. 实时业务数据通过受控工具查询，不直接连接业务数据库。
5. 人工客服可靠接管，接管期间 AI 不自动回复。
6. 新项目通过 Widget、API 和连接器快速接入。
7. 平台记录质量、用量、成本、同步和工具失败原因。

V1.0 先证明平台核心；博客是 V1.1 的首个内容源接入方，`go-mall` 是 V1.2 的首个实时工具接入方。两者都不是平台核心领域模型的一部分。

## 3. 当前技术决策

- 模块化单体，不提前拆微服务。
- FastAPI + Pydantic 2 + SQLAlchemy 2 Async + Alembic。
- PostgreSQL + pgvector 保存业务数据和早期向量索引。
- Redis + Celery 承担可靠后台任务。
- MinIO/S3 保存原始知识文件。
- Fake Chat/Embedding Provider + OpenAI-compatible Chat/Embedding Provider。
- 模型配置和租户 BYOK 保存在数据库，密钥加密。
- FastAPI/Pydantic 是 OpenAPI 单一事实来源，CI 保存和检查导出快照。
- 每个知识库固定 Embedding 模型和版本；RAG 使用向量与中文关键词混合检索。
- Content Source 和 Tool 是两类独立接入：文章等可索引内容进入 RAG，订单和库存等动态数据实时调用工具。
- 工具内部使用统一 JSON Schema 契约；先实现签名 HTTP，再增加 OpenAPI 导入和 MCP 适配。
- React + TypeScript 构建最小管理端和客服工作台，TypeScript 构建 Widget。
- 管理后台、客服工作台、中立示例站点和 Widget 支持英文与简体中文，浏览器可切换并持久化语言。

## 4. 开发阶段

### 阶段 A：V1.0 多租户 AI 客服核心

预计 25 个有效工作日。详细任务和验收见
[`v1-development-plan.md`](v1-development-plan.md)。

核心交付：

- 多租户、应用、认证、模型配置、会话和 SSE。
- 文件知识库、混合检索、引用、拒答和固定评估集。
- Widget、最小管理后台、人工客服工作台和人工接管。
- 管理后台、客服工作台、中立示例站点和 Widget 的英文/简体中文切换。
- 两个中立演示租户、隔离测试、部署和恢复文档。

本阶段不接入博客、商城或真实业务工具。只有 V1.0 自身验收全部通过，才算该版本完成。

### 阶段 B：V1.1 通用内容源与博客接入

预计 7 到 10 个有效工作日：

- 建立 Content Source 通用契约和同步状态模型。
- 支持全量、增量、幂等重试、下线传播和来源追踪。
- 接入博客文章并完成真实 Widget 闭环。
- 验收同步一致性、引用质量、故障恢复和跨租户隔离。

### 阶段 C：V1.2 通用只读工具与商城接入

预计 8 到 12 个有效工作日：

- 建立 Tool Definition、Binding、Invocation 和统一 JSON Schema 契约。
- 实现签名 HTTP、身份注入、防重放、超时、降级和审计。
- 接入 `go-mall` 的 `product.search` 和 `order.get`。
- 验收工具选择、Schema、用户归属、签名安全和商城真实 E2E。

### 阶段 D：V1.3 标准化连接器

预计 8 到 12 个有效工作日：

- 按白名单导入 OpenAPI `operationId`。
- 实现 MCP 客户端适配。
- 提供 Go、Python 连接器验证示例。
- 验收不同传输对统一工具契约的兼容性和安全边界。

### 阶段 E：V2.0 可运营和可收费产品

只有存在真实试用、预算或私有部署需求后才冻结范围。候选能力包括套餐、额度、用量对账、账单、白标、运营报表和私有部署工具。

多渠道、完整工单、SLA、技能组和高风险写工具是否进入 V2.0，由真实需求决定，不能预先全部承诺。

各阶段的完整通过标准见 [`version-roadmap.md`](version-roadmap.md)。后续版本必须同时通过既有版本回归测试。

## 5. 产品决策门禁

版本验收和产品决策分开执行：

| 观察结果 | 产品决定 |
| --- | --- |
| 博客需要稳定同步并用于客服 | 优先启动 V1.1 |
| 商城或客户需要订单等实时查询 | 优先启动 V1.2 |
| 接入方已有大量 OpenAPI 或 MCP 服务 | 启动 V1.3 |
| 客户认可回答质量，但接入困难 | 优先修正 SDK、Widget 和文档 |
| 知识问答召回不足 | 先优化数据清洗、检索和评估 |
| 人工接管使用频繁 | 单独规划队列、技能组和工单版本 |
| 有私有部署需求和预算 | 冻结并启动 V2.0 |
| 没有客户愿意付费 | 停止商业化扩张，保留已完成版本作为内部或项目交付能力 |

产品证据可以改变后续路线，但不能把已通过技术验收的版本重新定义为“未完成”。

## 6. 当前下一步

1. 冻结包含普通会话审阅、知识文档生命周期、间接 Prompt Injection 防护和双语修正的新 V1.0 候选。
2. 对同一候选重新执行自动化、OpenAPI、空库迁移、真实 PostgreSQL/Redis/MinIO/Worker 组件、RAG 固定集、
   真实 Provider、最终镜像 Secret、性能和灾备验收，不继承旧候选结果。
3. 由未参与平台开发的接入人依据 [`widget-onboarding-acceptance.md`](widget-onboarding-acceptance.md)
   完成一次 30 分钟内的中立 Web 接入，并由验收负责人在管理端核对 Conversation UUID。
4. 由独立评审人完成当前候选的 30 条人工复核，合并后使用 `--enforce-gate` 重新计算发布门禁。
5. 配置 Git Remote 并让同一候选执行云端 CI。以上门禁全部通过后更新验收结论并发布 `v1.0.0` Tag；
   此前不启动下一版本开发。
