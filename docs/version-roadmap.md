# AI 客服平台版本路线与逐版验收

## 1. 版本验收原则

版本验收只回答一个问题：该版本承诺的功能、质量、安全和文档是否全部完成。

- 阶段或每周检查只是开发过程中的质量反馈，不是版本验收，也不产生正式验收结论。
- 只有当前版本范围内的开发任务全部完成后，才开始执行该版本的正式验收。
- 验收对象必须固定到具体 Commit、构建产物和验收环境；开发中的滚动代码不能作为正式验收对象。
- 每个版本开发前固定目标、范围、明确不做、测试矩阵和量化门槛。
- 已进入开发的版本不得用后续版本功能代替当前版本缺失项。
- 当前版本验收通过后即视为完成；访谈数量、试用意愿和付费意愿不改变该结论。
- 产品验证只决定是否启动、调整或取消下一版本。
- 后续版本必须运行所有已发布版本的回归测试，不能以新功能破坏旧能力。
- 未通过验收的版本不得打正式 Tag，也不得在文档中标记为已发布。

每次正式验收必须保存一份独立记录，至少包含：

```text
版本号和验收日期
Git Commit 和构建产物标识
验收环境与依赖版本
数据库迁移与回滚结果
测试命令、测试结果和失败说明
RAG 或连接器评估数据集版本与指标
安全、隔离和性能结果
已知限制与遗留问题
最终结论：通过 / 不通过
验收人
```

验收记录保存在 `docs/acceptance/<version>.md`，格式见
[`acceptance/README.md`](acceptance/README.md)。自动化报告保存在 CI Artifact，不把真实用户数据或密钥提交到仓库。

## 2. 版本顺序

| 版本 | 核心目标 | 预计开发时间 | 主要真实接入 |
| --- | --- | ---: | --- |
| V1.0 | 多租户 AI 客服核心 | 25 个有效工作日 | 中立演示站点 |
| V1.1 | 通用内容源与知识同步 | 7 到 10 个有效工作日 | 博客项目 |
| V1.2 | 通用只读工具连接器 | 8 到 12 个有效工作日 | `go-mall` |
| V1.3 | 标准化连接器接入 | 8 到 12 个有效工作日 | OpenAPI、MCP、SDK 示例 |
| V2.0 | 可运营和可收费产品 | 需求冻结后估算 | 真实试用租户 |

开发时间只是排期参考，不是完成标准。每个版本只能由对应验收门槛判定完成。

## 3. 所有版本的共同发布门禁

每个版本都必须满足以下基础条件：

- Ruff、mypy、后端测试、前端类型检查和该版本 E2E 全部通过。
- Alembic 空库迁移、上一正式版本升级和回滚方案验证通过。
- OpenAPI 快照不存在未说明的破坏性变化。
- 跨租户读取、修改、删除、缓存、对象存储、任务和向量检索成功数为 `0`。
- Git、日志、错误响应、测试报告和构建产物中的真实 Secret 数为 `0`。
- Debug、默认密码和匿名管理入口在发布环境中关闭。
- 部署、升级、回滚、配置和已知限制文档已更新。
- 生成该版本验收记录，所有未通过项有明确结论，不能用“后续再测”判定通过。

## 4. V1.0：多租户 AI 客服核心

### 4.1 开发范围

- Tenant、Application、管理员认证、应用凭据和短期 Customer Token。
- 后台管理的 Fake 与 OpenAI-compatible Chat/Embedding Provider。
- 会话、消息、SSE 流式回复、用量和审计。
- 文件知识库、异步入库、混合检索、引用、拒答和评估集。
- JavaScript SDK、Web Widget、最小管理后台和人工客服工作台。
- 用户申请转人工、原子接管、人工回复和关闭。
- Docker Compose、CI、备份恢复和单机部署文档。

V1.0 不接入博客、商城或任何真实业务工具。详细任务见
[`v1-development-plan.md`](v1-development-plan.md)。

### 4.2 版本验收

- 两个知识内容不同的演示租户可以独立创建应用、知识库和会话。
- 跨租户访问成功数为 `0`，隔离测试覆盖 API、数据库、Redis、对象存储、任务和向量检索。
- 150 条固定评估集达到 V1.0 计划中的 RAG、拒答和转人工指标，严重错误回答为 `0`。
- 中立 Web 页面只依据文档，在 30 分钟内完成 Widget 接入和一次有效对话。
- PC 与 375px 移动宽度完成对话、引用、拒答、申请转人工、客服接管和关闭 E2E。
- 20 个并发 SSE 会话没有消息串流、状态错乱或连接泄漏。
- 人工接管期间 AI 自动回复数为 `0`，两名客服不能同时接管同一会话。
- 空库部署、上一迁移升级、备份、恢复和回滚演练全部通过。
- 保存 `docs/acceptance/v1.0.md` 后才允许发布 `v1.0.0` Tag。

## 5. V1.1：通用内容源与博客接入

### 5.1 开发范围

- 新增 `Integration`、`ContentSource`、`SyncJob` 和来源文档映射。
- 定义通用内容源协议，支持全量同步、增量游标、幂等重试和删除/下线传播。
- 第一种传输使用受控 HTTP 拉取；博客提供结构化文章接口并完成真实接入。
- 同步后的文章进入现有知识库流程，保留原始来源 ID、URL、更新时间和版本。
- 后台可创建、测试、启停内容源，并查看同步进度、错误和最近成功时间。
- 博客页面接入 Widget，回答可引用真实文章页面。

V1.1 不实现实时业务工具、商城接入、写操作、OpenAPI 自动导入或 MCP。

### 5.2 版本验收

- 对测试数据集中全部可发布文章执行全量同步，源端与平台有效文档数量一致。
- 相同全量同步连续执行两次，重复文档和重复有效版本数量均为 `0`。
- 新增、修改、下线和删除各类增量事件全部正确传播，成功率为 `100%`。
- Worker 在同步中断后可以幂等重试或从游标恢复，不产生半发布文档。
- 每个同步文档都能追溯到内容源、外部 ID、来源 URL 和源更新时间。
- 博客问答评估中的引用可打开且支持回答的比例不低于 `95%`。
- 内容源凭据、同步任务和文档的跨租户访问成功数为 `0`。
- 博客真实环境完成“发布文章 -> 同步 -> Widget 提问 -> 引用原文”E2E。
- V1.0 全部回归测试继续通过，并保存 `docs/acceptance/v1.1.md`。

## 6. V1.2：通用只读工具连接器与商城接入

### 6.1 开发范围

- 新增 `Integration`、`ToolDefinition`、`ToolBinding` 和 `ToolInvocation`。
- 工具输入输出使用 JSON Schema 2020-12，模型只生成允许暴露的业务参数。
- 第一种传输实现签名 HTTP，支持时间戳、Nonce、防重放、超时、重试和审计。
- 租户、应用和最终用户身份由平台服务端注入，禁止接受模型生成的身份覆盖。
- `go-mall` 实现 `product.search` 和 `order.get` 两个只读工具。
- 工具失败、超时、无权限或结果不确定时明确降级或转人工。
- 后台支持工具注册、测试、绑定、启停和调用审计。

V1.2 不允许自动退款、改地址、取消订单等写操作，也不实现 OpenAPI 自动导入或 MCP。

### 6.2 版本验收

- 不少于 30 条工具选择评估用例，工具选择正确率不低于 `95%`。
- 非法输入、非法输出和未声明字段被 Schema 校验拒绝的比例为 `100%`。
- 当前登录商城用户可以查询本人订单，跨用户和跨租户订单查询成功数为 `0`。
- 过期时间戳、重复 Nonce、错误签名和被停用凭据的调用成功数为 `0`。
- 连接器超时、5xx、非法响应和不可达场景均返回明确错误，不伪造业务结果。
- 工具调用审计覆盖率为 `100%`，包含租户、应用、会话、工具、耗时、状态和脱敏参数摘要。
- `go-mall` 完成“商品提问 -> 工具查询”和“本人订单提问 -> 授权查询”真实 E2E。
- V1.0、V1.1 全部回归测试继续通过，并保存 `docs/acceptance/v1.2.md`。

## 7. V1.3：OpenAPI、MCP 与连接器 SDK

### 7.1 开发范围

- 从 OpenAPI 3.1 文档中只导入管理员明确勾选的 `operationId`，转换为平台统一工具契约。
- 实现 MCP 客户端适配层，把允许的远程工具映射到同一 `ToolDefinition`。
- 提供最小连接器验证 SDK 和示例，帮助 Go、Python 服务验证签名、时间戳和 Nonce。
- 统一 Signed HTTP、OpenAPI 和 MCP 的超时、权限、审计、错误和结果规范化行为。

V1.3 不建设公开连接器市场，不允许未经审核自动暴露整份业务 API，也不默认开放写工具。

### 7.2 版本验收

- 示例 OpenAPI 文档可成功解析，只有白名单 `operationId` 被导入，未选择接口暴露数为 `0`。
- 缺少唯一 `operationId`、不支持的 Schema 和不安全外部引用会被明确拒绝。
- OpenAPI URL、重定向和引用解析通过 SSRF 测试，内网和云元数据地址访问成功数为 `0`。
- MCP 连接完成协议协商、工具发现和工具调用契约测试，未知或被停用工具不可调用。
- 同一个标准工具通过 Signed HTTP、OpenAPI 和 MCP 三种适配后得到等价的规范化结果。
- Go 和 Python 示例均通过签名正确、签名错误、过期和重放测试。
- 所有适配方式继续满足 V1.2 的身份注入、权限和审计要求。
- V1.0 至 V1.2 全部回归测试继续通过，并保存 `docs/acceptance/v1.3.md`。

## 8. V2.0：可运营和可收费产品

### 8.1 立项前先冻结范围

V2.0 只有在存在明确试用或付费需求后立项。候选范围包括：

- 套餐、额度、用量对账、超额限制和账单明细。
- 租户自助开通、升级、停用、数据导出和删除。
- 白标配置、运营报表、告警和更完整的审计。
- 私有部署安装、升级、备份、恢复和支持流程。

在线支付、发票、多渠道、完整工单、SLA 和技能组不得默认全部塞入 V2.0；立项时只选择有真实需求的一组能力，并补充接口和数据模型。

### 8.2 最低版本验收

- 冻结范围中的每个套餐、额度和状态转换都有正常、边界、并发和回滚测试。
- 固定用量数据集的明细、聚合、额度扣减和账单结果完全一致，可追溯到原始用量记录。
- 达到额度、超额、降级、停用和恢复场景按契约执行，不发生跨租户额度影响。
- 管理员敏感操作全部有审计，租户数据导出和删除经过权限与完整性验收。
- 私有部署在一台全新目标服务器上只依据文档完成安装、升级、备份和恢复。
- 压力测试、故障恢复和容量结果达到立项时冻结的目标。
- V1.0 至 V1.3 全部回归测试继续通过，并保存 `docs/acceptance/v2.0.md`。

## 9. 产品决策门禁

产品决策在版本验收之外单独记录：

| 证据 | 影响的决定 |
| --- | --- |
| 博客需要持续自动同步文章 | 是否优先启动 V1.1 |
| 商城或客户需要实时订单查询 | 是否优先启动 V1.2 |
| 接入方已有大量 OpenAPI/MCP 服务 | 是否启动 V1.3 |
| 有稳定试用、预算或私有部署合同 | 是否立项 V2.0 |
| 没有付费意愿 | 停止商业化扩张，但不否定已完成版本 |

未列入本路线的多渠道、完整工单、SLA、技能组和高风险写工具，必须在立项时新建版本章节并先写验收标准，再进入开发。
